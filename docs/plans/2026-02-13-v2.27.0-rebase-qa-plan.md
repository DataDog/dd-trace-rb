# QA Plan: dd-trace-rb v2.15.0 → v2.30.0 Rebase (PR #30)

## Context

Braze maintains a fork of DataDog/dd-trace-rb that originally carried 3 semantic patches, carried forward across 6 upstream merge cycles since 2019. PR #30 rebases the fork from v2.15.0 to v2.30.0. All **3 surviving patches** are carried forward (Redis instrumentation enhancements, Grape endpoint namespace fix, and ActiveSupport::Cache bypass). This plan covers validation, test coverage, and engineer follow-up for all three patches.

**Update (2026-02-20):** The Grape endpoint namespace patch (Patch #2) was initially removed during the v2.27.0 rebase, believing upstream handled it correctly. Platform test `api_datadog_correctness_spec.rb` proved this was wrong — upstream's `api_view` calls `api.base.to_s` which does not return the correct namespace for Braze's API structure. The patch has been restored. The fork was also rebased from v2.27.0 to v2.29.0 (none of the patched files changed between these versions), and subsequently to v2.30.0 (only the Grape `api_view` method changed upstream — added a `respond_to?(:base)` guard for Grape >= 2.3.0; the Braze caller-level bypass is unaffected).

### 1B. Grape Endpoint Naming (Patch #2) — Carry Forward (Restored)

**Contact:** Matt DiSipio, @Appboy/monolith-foundations (prior contributor: Zach McCormick)

#### Investigation findings

The Grape `api_view` workaround was added by Matt DiSipio in April 2022 (MOPS-596) to fix endpoint naming broken by [ruby-grape/grape#1825](https://github.com/ruby-grape/grape/issues/1825). In Grape >= 1.2.0, `class Foo < Grape::API` creates an internal anonymous `Instance` subclass, and `endpoint.options[:for]` returns that Instance class. Calling `.to_s` or `.name` on it returned `nil` or an anonymous class representation instead of `"Foo"`.

**The issue was fixed upstream in Grape** via [PR #1826](https://github.com/ruby-grape/grape/pull/1826), which overrides `.to_s` on the Instance class to delegate to `base.to_s`, returning the user-visible class name. **Braze is on Grape 1.8.0**, well past this fix.

**The upstream dd-trace-rb `api_view` (v2.27.0) also handles this correctly:**
```ruby
def api_view(api)
  if defined?(::Grape::API::Instance) && api <= ::Grape::API::Instance
    api.base.to_s  # returns "InternalAPI::Base" etc.
  else
    api.to_s
  end
end
```

Both the Braze patch (`api.to_s` with blank fallback) and the upstream code (`api.base.to_s`) produce identical results with Grape 1.8.0. Braze's Grape APIs all use standard `class Foo < Grape::API` inheritance — no unusual mounting patterns.

#### Action: Carry forward the patch (restored after test failure)

**Update (2026-02-20):** The patch was initially removed during the v2.27.0 rebase based on the analysis above. However, platform test `api_datadog_correctness_spec.rb` proved that upstream's `api_view` does NOT produce correct namespaces for Braze's API structure. The upstream helper calls `api.base.to_s` which does not return the expected full class name (e.g. `Appboy::External::API::Campaigns::Service`). The Braze patch calls `api.to_s` directly, which correctly returns the full namespace. The patch has been restored in both `endpoint_start_process` and `endpoint_run`.

**Update (v2.30.0 rebase):** Upstream v2.30.0 added a `respond_to?(:base)` guard to `api_view` for Grape >= 2.3.0 compatibility. The Braze patch bypasses `api_view` at the caller level (calling `api.to_s` directly instead), so this upstream change has no effect on the patch's behavior or correctness.

### 1C. ActiveSupport::Cache Bypass (Patch #3) — Carry forward

**Contact:** Evan Goldenberg, @Appboy/product-sre, @Appboy/monolith-foundations (owns `redis-rails` in Gemfile)

#### Investigation findings

We investigated Braze's caching architecture to understand why this patch exists and whether it can be removed. Key findings:

**Braze's caching does NOT use `ActiveSupport::Cache::RedisStore`:**

| Cache layer | Underlying store | Uses `ActiveSupport::Cache`? |
|---|---|---|
| `Appboy::Cache` | `ActiveSupport::Cache::MemCacheStore` (memcached) | Yes — the only AS::Cache backend |
| `Braze::Core::Caches::RedisAdapter` | Direct Redis pool (`redis-rb`) | No |
| `RedisProvider.all_cache_shards()` | Direct Redis pool (`ha_cache_urls`) | No |
| `Braze::Core::Caches::CascadingCacheAdapter` | Chains local → `CacheAdapter` (memcached) | Indirectly via `Appboy::Cache` |
| `Braze::Core::Caches::InMemoryCacheAdapter` | `LruRedux::TTL::ThreadSafeCache` | No |
| `Appboy::LocalRemoteCacheAdapter` | `Appboy::MachineCache` + `Appboy::Cache` | Indirectly via `Appboy::Cache` |

Redis is heavily used for caching, but through Braze's own abstractions (`RedisProvider` cache shards, `RedisAdapter`) that use raw `redis-rb` connections — never through `ActiveSupport::Cache::RedisStore`.

**Why `redis-activesupport` is in the bundle at all:**

```
Gemfile: gem "redis-rails"     # @owner @Appboy/monolith-foundations
  → redis-actionpack            # NEEDED: provides :redis_store session backend for dashboard
  → redis-activesupport         # NOT NEEDED: defines ActiveSupport::Cache::RedisStore
  → redis-store                 # shared base
```

`redis-rails` is a meta-gem used for the **dashboard session store** (`session_store.rb` configures `:redis_store` via `redis-actionpack`). The `redis-activesupport` half is unused baggage, but it causes `ActiveSupport::Cache::RedisStore` to exist at load time, which triggers dd-trace-rb's `patch_redis_store?` detection.

**No code in the platform references `ActiveSupport::Cache::RedisStore`** — confirmed by grep across the entire repository. `redis-rails` has been in the Gemfile since the repo's initial commit (July 2014) and `redis-activesupport` was never used.

#### How the bug manifested without ever using `RedisStore`

The bug was a "phantom dependency" problem: `redis-activesupport` was never used, but its mere presence in the bundle changed dd-trace-rb's boot-time patching decisions, causing instrumentation to be prepended onto the wrong class.

On the dd-trace-rb version the fork was running (≤ [v2.1.0](https://github.com/DataDog/dd-trace-rb/tree/v2.1.0)), the legacy ActiveSupport::Cache patcher worked as follows:

1. [`cache_store_class(meth)`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/redis.rb#L31-L37) returned **a single class** — either `RedisStore` or `Store`, not both. When [`patch_redis?(meth)`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/redis.rb#L25-L29) was true, it returned `RedisStore`; otherwise it fell through to [`super`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/patcher.rb#L34-L36) which returned `Store`.
2. `patch_redis?` checked [`Gem.loaded_specs['redis-activesupport']`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/redis.rb#L26) and [`defined?(::ActiveSupport::Cache::RedisStore)`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/redis.rb#L27) — both true because `redis-activesupport` was in the bundle — and whether [`RedisStore.instance_methods(false).include?(meth)`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/redis.rb#L28). No check was made for whether `RedisStore` was actually instantiated or configured.
3. For methods that `RedisStore` overrides (e.g., `:write`, `:read_multi`, `:fetch_multi` — see [redis-activesupport source](https://github.com/redis-store/redis-activesupport/blob/v5.3.0/lib/active_support/cache/redis_store.rb)), `patch_redis?` returned true, so [`patch_cache_store_write`](https://github.com/DataDog/dd-trace-rb/blob/7b867f6f9b1c13e4098eed9274e62d8f23b286c1/lib/datadog/tracing/contrib/active_support/cache/patcher.rb#L56-L58) called `RedisStore.prepend(Instrumentation::Write)` — prepending instrumentation **only onto `RedisStore`**, not `Store`.
4. Since `MemCacheStore` inherits from `Store` (not `RedisStore`), calls to `MemCacheStore#write` **skipped the instrumentation entirely**. The phantom class definition hijacked the monkey-patching target.

**The upstream fix:** In [dd-trace-rb v2.2.0](https://github.com/DataDog/dd-trace-rb/releases/tag/v2.2.0) (July 11, 2024), [PR #3772](https://github.com/DataDog/dd-trace-rb/pull/3772) by Marco Costa rewrote the patcher to:
- Return an **array** of classes (`[RedisStore, Store]`) instead of a single class, ensuring `Store` is always prepended regardless of whether `RedisStore` exists
- Reduce the legacy-patched methods from 7 to 3 (`:read_multi`, `:fetch`, `:fetch_multi`), with the rest now handled by the new event-based instrumentation path

This independently fixes the phantom dependency bug — even with `redis-activesupport` loaded, `Store` always gets prepended. However, the cleanest fix is still to remove the phantom dependency entirely (which is what we're doing).

#### Action: Carry forward the patch

We are keeping the `patch_redis_store? = false` override in the fork. While we investigated replacing `redis-rails` with `redis-actionpack` in the platform Gemfiles (which would remove the phantom dependency entirely), the QA overhead of changing gems in the platform is not justified for a 5-line patch that:

- Is stable and has been carried forward through multiple rebase cycles
- Will be eliminated automatically when dd-trace-rb 3.0 removes the entire legacy monkey-patching code path (see Appendix B)
- Is independently made safe by the upstream v2.2.0 fix (PR #3772), which changed `cache_store_class` to return an array — so even without our patch, `Store` is always prepended

The `redis-rails` → `redis-actionpack` option is documented here for future reference if the team decides the Gemfile cleanup is worth doing independently.

## Phase 2: Automated Tests

Based on engineer feedback from Phase 1, write the following tests. These live in the dd-trace-rb fork.

### Test Organization Strategy

All Braze-specific tests live in a single dedicated directory tree, **separate from upstream specs**:

```
spec/
  braze/                          # ← All Braze-owned tests live here
    contrib/
      redis/
        thread_local_tags_spec.rb
        trace_middleware_metrics_spec.rb
    spec_helper.rb                # Braze-specific shared setup (thread-local cleanup, etc.)
```

Note: No `grape/` test directory is needed — Patch #2 is being removed (upstream handles it with Grape 1.8.0; see Phase 1B). An `active_support/` test directory is not strictly required since the patch is a simple `return false`, but a test confirming the upstream detection would return `true` (proving the patch is still needed) could be added in the future.

**Why this structure matters for rebases:**

1. **No merge conflicts in spec/.** Upstream never touches `spec/braze/`, so rebases are conflict-free for tests. This is the same principle as organizing patches into semantic commits — the test directory mirrors the patch grouping.

2. **Easy to identify Braze-owned code.** `git diff --stat upstream/main` will show `spec/braze/` as a clear, self-contained block. A future maintainer running `find spec/braze` immediately knows the full scope of custom test coverage.

3. **Survives upstream spec reorganizations.** If upstream renames `spec/datadog/tracing/contrib/redis/` or restructures its test layout, Braze tests are unaffected.

4. **Maps 1:1 to patch inventory.** Each subdirectory under `spec/braze/contrib/` corresponds to a row in the Patch Inventory table. When evaluating whether a patch can be dropped, you also know exactly which tests to remove.

5. **Dedicated commit.** Tests should be added in their own commit (e.g., `[STREAM-126] Add Braze patch QA tests`) so they can be cherry-picked or dropped independently of the patches themselves during future rebases.

Each spec file should include a header comment linking back to the patch it validates:

```ruby
# frozen_string_literal: true

### BRAZE TEST — validates Patch #1 (Redis instrumentation enhancements)
### See: docs/plans/2026-02-13-v2.27.0-rebase-qa-plan.md
### Patched files: lib/datadog/tracing/contrib/redis/{ext,tags,trace_middleware}.rb
```

require 'datadog/tracing/contrib/redis/tags'

    end

  it 'sets redis.codeowner from thread-local' do
    Thread.current[:redis_operation_codeowner] = '@Appboy/in-memory-data'
    Datadog::Tracing::Contrib::Redis::Tags.set_common_tags(client, span, raw_command)
    expect(span.get_tag('redis.codeowner')).to eq('@Appboy/in-memory-data')
  end

  it 'sets redis.shard_index from thread-local' do
    Thread.current[:redis_operation_shard_index] = '3'

## Appendix A: Upstream Proposal Details

### Redis Custom Tag Callback

**Problem:** Braze needs to attach application-level metadata (calling file path, code owner, shard index, company name) to Redis spans at trace time. The only way to do this today is by forking dd-trace-rb and modifying `Tags.set_common_tags`.

**Proposed solution:** Add an optional `on_trace` callback to Redis contrib configuration:

```ruby
Datadog.configure do |c|
  c.tracing.instrument :redis, on_trace: ->(span, client, command) {
    span.set_tag('redis.filepath', Thread.current[:redis_operation_filepath])
    span.set_tag('redis.codeowner', Thread.current[:redis_operation_codeowner])
    span.set_tag('redis.shard_index', Thread.current[:redis_operation_shard_index])
    span.set_tag('company_name', Thread.current[:redis_operation_company_name])
    span.set_tag('redis.key', command[1]) if command.is_a?(Array) && command.size > 1
    span.set_metric('redis.raw_command_length', command.to_s.length)
  }
end
```

#### Why existing extension points are insufficient

We evaluated every extension mechanism in dd-trace-rb v2.27.0 to see if any could replace Patch #1 without forking. The patch does three things:

| Part | What it does | Could be done without forking? |
|------|-------------|-------------------------------|
| A | Thread-local tags (`redis.filepath`, `redis.codeowner`, `redis.shard_index`, `company_name`) | **Partially** — `Datadog::Tracing.active_span` can set tags from application code, but requires restructuring the platform's `trace_with_custom_tags` to tag the span directly instead of setting thread-locals |
| B | `redis.key` extraction from `command[1]` | **No** — requires access to the `command` argument inside `TraceMiddleware.call` |
| C | `redis.raw_command_length` / `redis.raw_response_length` metrics | **No** — requires wrapping `yield` inside the middleware to capture the return value and measure sizes |

**Extension points evaluated:**

1. **`Datadog::Tracing.before_flush`** — Runs in the TraceWriter's background thread, not the application thread. Thread-local variables are not available. Can only modify spans based on data already on the span.
2. **`Datadog::Tracing.active_span`** — Works for Part A (tags), but doesn't help with Parts B or C since we have no access to the command array or the yield result from outside the middleware.
3. **OpenTelemetry SpanProcessor** — Only intercepts spans created via the OTel API. Redis contrib uses the Datadog tracer directly, so OTel processors never see these spans.
4. **Redis contrib configuration** — Has `enabled`, `service_name`, `command_args`, `peer_service`, and analytics settings. No callback hooks (`on_trace`, `before_trace`, `on_error`, etc.).
5. **`on_error` callback** — Some contribs (Sidekiq, Que, MySQL2) support this, but it only fires on errors. Redis contrib doesn't support it at all.

**Conclusion:** An upstream `on_trace` callback is the only way to eliminate the fork patch. It would need to provide `(span, client, command)` and fire inside the trace block (before yield for pre-tags, or wrapping yield for response measurement). This is a clean extension point that doesn't change default behavior and would benefit any user needing custom Redis span enrichment.

## Appendix B: dd-trace-rb 3.0 and the ActiveSupport::Cache Patch Sunset

### How ActiveSupport::Cache instrumentation works today (v2.x)

In dd-trace-rb 2.x, ActiveSupport::Cache is instrumented through **two parallel strategies** that run simultaneously:

1. **Event-based (the new approach):** `Events::Cache.subscribe!` subscribes to `ActiveSupport::Notifications` events like `cache_read.active_support`, `cache_write.active_support`, etc. This subscriber has a `trace?` method (`cache/events/cache.rb:48-67`) that can filter by cache store class at runtime via the `cache_store` configuration option. This is the preferred, forward-looking approach.

2. **Legacy monkey-patching (the old approach):** `Patcher.patch_legacy_cache_store` (`cache/patcher.rb:47-51`) prepends instrumentation modules (`Instrumentation::Fetch`, `Instrumentation::ReadMulti`, `Instrumentation::FetchMulti`) directly onto cache store classes at boot time. The `cache_store_class` method (`cache/redis.rb:48-56`) determines *which* classes to prepend onto, and `patch_redis_store?` controls whether `ActiveSupport::Cache::RedisStore` (from the `redis-activesupport` gem) is included.

**Our patch** hard-codes `patch_redis_store?` to return `false` (`cache/redis.rb:26-30`) because Braze doesn't use Redis through `ActiveSupport::Cache`. Without the patch, the legacy monkey-patching would prepend instrumentation onto `RedisStore`, which Braze's comment says "causes the wrong `Store` to be patched."

### What the DEV-3.0 annotations say

The dd-trace-rb codebase contains extensive `DEV-3.0` comments (14 instances across 3 files) that explicitly plan to remove the legacy monkey-patching layer:

**`cache/instrumentation.rb:13-35`** — The authoritative comment block:
> DEV-3.0: `ActiveSupport::Cache` is now instrumented by subscribing to ActiveSupport::Notifications events. The implementation is located at `Events::Cache`. The events emitted provide richer introspection points (e.g. events for cache misses on `#fetch`) while also ensuring we are using Rails' public API for improved compatibility.
>
> But a few operations hold us back:
> 1. `#fetch` produces nested spans in 2.x vs. sibling spans in the event-based approach (breaking change in span tree structure)
> 2. `#read_multi` / `#fetch_multi` events were only introduced in ActiveSupport 5.2.0
>
> At the end of the day, moving to ActiveSupport events is the better approach, but we have to retain this last few monkey patches (and all the supporting code) to avoid a breaking change for now.

**`cache/patcher.rb:34-36`** — Tags `patch_legacy_cache_store` for removal.

**`cache/events/cache.rb:57-59`** — Tags the backwards-compatible `nested_read?`/`nested_multiread?` checks for removal. Once the legacy layer is gone, these guard checks become unnecessary.

### What this means for Braze's patch

When dd-trace-rb 3.0 is released:

| Component | Current (2.x) | After 3.0 |
|-----------|---------------|-----------|
| `patch_legacy_cache_store` | Called at boot, prepends instrumentation | **Removed entirely** |
| `cache_store_class` / `patch_redis_store?` | Controls which classes get monkey-patched | **Removed entirely** |
| `Instrumentation::Fetch`, `ReadMulti`, `FetchMulti` | Monkey-patch modules prepended onto store classes | **Removed entirely** |
| `Events::Cache.subscribe!` | Runs alongside legacy patching | **Sole instrumentation mechanism** |
| `cache_store` config option | Filters event-based spans by store class | Still available (and now the only filtering mechanism) |

**Result:** Braze's `patch_redis_store? = false` patch will have nothing left to modify. The entire code path it short-circuits will no longer exist.

**Note:** We are carrying the patch forward until dd-trace-rb 3.0 removes the legacy code path. The patch is 5 lines, stable, and lower-risk than changing the platform's `redis-rails` dependency. The upstream v2.2.0 fix (PR #3772) also independently addresses the underlying bug, making our patch a belt-and-suspenders safeguard on v2.27.0.
